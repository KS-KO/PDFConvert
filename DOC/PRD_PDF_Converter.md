# PDF/Image to Office/Slides Converter PRD

## 1. 문서 개요

- 문서명: PDF 및 이미지를 PPT, Word, Google Slides 및 DOC로 변환하는 프로그램 PRD
- 문서 버전: v0.6
- 작성일: 2026-04-07
- 문서 목적: PDF 또는 이미지 파일을 편집 가능한 문서 및 프레젠테이션 형식으로 변환하는 Windows 데스크톱 애플리케이션의 제품 요구사항과 현재 구현 상태를 정의한다.

## 2. 제품 개요

본 프로그램은 Windows x64 환경에서 실행되는 WPF 기반 데스크톱 애플리케이션으로, 사용자가 선택한 PDF 또는 이미지 파일을 `PPTX`, `DOCX`, `DOC`, `Google Slides` 대상 형식으로 변환하는 것을 목표로 한다.

현재 구현은 `.NET 9`, `WPF`, `MVVM` 기준으로 진행되었으며, MVP 범위 내에서 `PDF/Image -> PPTX`, `PDF/Image -> DOCX`, `PDF/Image -> DOC` 변환 흐름, 결과 확인 UI, 최근 이력 저장/복원, OCR 엔진 선택, 루트 `bin\Debug`, `bin\Release` 기준 빌드 출력 구조까지 포함한다.

## 3. 기술 사양 및 제약사항

### 필수 기술 사양

- 개발 런타임: `.NET 9`
- 빌드 아키텍처: `x64 전용`
- UI 프레임워크: `WPF`
- 아키텍처 패턴: `MVVM`

### 기술 제약사항

- Windows 데스크톱 앱으로만 제공한다.
- `Any CPU`, `x86`, `ARM` 빌드는 범위에서 제외한다.
- 빌드 및 배포 산출물은 가능한 한 단일 실행 폴더 중심으로 단순화해야 한다.
- 불필요한 중간 산출물, 중복 라이브러리 폴더 구조, 복잡한 하위 폴더 생성을 최소화해야 한다.
- 개발 및 검수 편의를 위해 빌드 산출물은 `bin\Debug`, `bin\Release`처럼 구성별로 구분되어야 한다.
- View는 표시와 입력 처리에 집중하고, 로직은 ViewModel 및 서비스 계층으로 분리한다.
- 변환 엔진, 파일 저장, 외부 연동, 이력 저장은 테스트 가능한 서비스 계층으로 구성한다.

## 4. 제품 목표

- PDF 및 이미지 파일을 `PPTX`, `DOCX`, `DOC`, `Google Slides` 형식으로 변환할 수 있는 통합 경험 제공
- 텍스트, 문단, 제목, 목록 구조를 최대한 보존
- 비전문 사용자도 쉽게 사용할 수 있는 단순한 데스크톱 UI 제공
- 진행 상태, 결과 파일, 경고 메시지, 최근 변환 이력을 확인 가능한 UX 제공
- 향후 OCR, Google Slides API 직접 연동, 품질 개선 로직으로 확장 가능한 구조 확보

## 5. 문제 정의

사용자는 PDF 및 이미지 내용을 재활용하는 과정에서 다음과 같은 문제를 겪는다.

- 내용 복사/재작성하며 수작업으로 다시 문서를 만들어야 한다.
- 변환 도구가 문단 구조, 제목, 목록, 페이지 단위를 충분히 보존하지 못한다.
- 발표자료용 변환과 문서용 변환을 별도 도구로 처리해야 한다.
- 변환 결과를 다시 찾거나 반복 작업을 추적하기 어렵다.

- PDF 텍스트 추출 및 문단/목록/제목 후처리
- 이미지 파일 및 PDF 내 이미지 대상 OCR 기반 글자 추출 지원

### Out of Scope

- 고급 문서 편집기
- 모바일 앱
- 모든 도형, 벡터, 표의 완전한 재현
- Google Slides 직접 API 생성

## 9. 주요 사용 시나리오

1. 사용자가 프로그램을 실행한다.
2. 사용자가 PDF 또는 이미지 파일을 선택한다.
3. 사용자가 출력 형식과 출력 폴더를 선택한다.
4. 시스템이 파일을 분석하고 텍스트를 추출한다 (이미지의 경우 OCR 필수 수행).
5. 시스템이 선택한 포맷에 맞는 결과 파일을 생성한다.
6. 사용자는 진행 상태, 생성 파일 목록, 경고, 출력 폴더를 확인한다.
7. 사용자는 최근 변환 이력을 통해 이전 결과를 다시 확인한다.

## 10. 기능 요구사항

### 10.1 파일 입력

- 사용자는 단일 PDF 파일 또는 이미지 파일을 선택할 수 있어야 한다.
- 지원 이미지 형식: `.jpg`, `.jpeg`, `.png`, `.bmp`, `.gif`, `.tiff`
- 시스템은 최근 선택한 파일 경로를 기억하고 다음 실행 시 복원할 수 있어야 한다.
- 손상되었거나 읽을 수 없는 파일에 대해 오류 메시지를 제공해야 한다.

### 10.2 출력 형식 선택

- 사용자는 `PPTX`, `DOCX`, `DOC`, `Google Slides`를 선택할 수 있어야 한다.
- 복수 포맷 선택을 지원해야 한다.

### 10.2.1 Git 버전 정보 표시

- 프로그램은 Git 기반 버전 관리를 전제로 해야 한다.
- 상태표시바 우측에는 현재 빌드 기준 `Commit Count`와 `hash 9자리`를 표시해야 한다.
- Git 정보는 앱 버전 식별 및 사용자 지원 시점의 빌드 추적에 활용할 수 있어야 한다.
- Git 정보를 읽을 수 없는 환경에서는 빈 값 대신 대체 표시값을 제공해야 한다.

### 10.3 출력 경로 선택

- 사용자는 결과 저장 폴더를 선택할 수 있어야 한다.
- 최근 선택한 출력 폴더를 다음 실행 시 복원할 수 있어야 한다.
- 변환 후 출력 폴더를 바로 열 수 있어야 한다.

### 10.4 변환 엔진

- 시스템은 PDF에서 텍스트를 추출하거나, 이미지에서 OCR을 수행해야 한다.
- 이미지 파일이 입력된 경우, OCR 엔진을 통해 전체 텍스트와 좌표 정보를 추출한다.
- 사용자는 OCR 엔진을 선택할 수 있어야 하며, 최소 `기본 OCR`, `Tesseract`, `Google Vision OCR`, `Naver Clova OCR`를 지원 대상에 포함해야 한다.
- PDF 텍스트 레이어와 이미지 OCR 결과가 중복되는 경우 결과 문서에서는 중복 텍스트를 제거해야 한다.
- 단어 좌표 기반으로 줄 경계를 복원해야 한다.
- 큰 가로 간격이 있는 경우 간단한 컬럼 분리를 시도해야 한다.
- 수평 텍스트 위주로 추출해 세로 라벨 노이즈를 일부 줄여야 한다.
- 제목 후보, 문단, 목록 항목을 후처리 규칙으로 구분해야 한다.

### 10.4.1 OCR 엔진 선택

- 사용자는 OCR 사용 여부를 켜고 끌 수 있어야 한다 (이미지 파일의 경우 항상 활성화됨).
- OCR 사용 시 엔진 선택 UI를 통해 원하는 OCR 엔진을 지정할 수 있어야 한다.
- 지원 대상 OCR 엔진은 `기본 OCR`, `Tesseract`, `Google Vision OCR`, `Naver Clova OCR`를 포함해야 한다.
- 엔진별 언어 설정, 인증 정보, 품질 차이, 비용 여부를 고려할 수 있는 확장 구조여야 한다.
- 선택한 OCR 엔진을 사용할 수 없는 환경에서는 대체 엔진 또는 오류 안내를 제공해야 한다.

### 10.5 포맷별 요구사항

#### PPTX

- 입력 파일의 페이지(또는 이미지 파일당)별 1슬라이드 생성
- 제목과 본문 블록 분리
- 목록 항목은 별도 문단으로 유지

#### DOCX

- 페이지별 제목 삽입
- 문단 블록 단위로 본문 구성
- 목록 항목은 별도 문단으로 구성

#### DOC

- 호환 목적의 HTML 기반 `.doc` 파일 생성
- 제목, 문단, 목록을 단순 구조로 보존

#### Google Slides

- 현재는 직접 API 연동 대신 placeholder 안내 파일 생성

### 10.6 진행 상태 및 결과

- `대기`, `검증`, `분석`, `변환`, `완료`, `실패` 상태를 표시해야 한다.
- 생성된 파일 목록을 화면에 보여줘야 한다.
- 경고 메시지를 사용자에게 표시해야 한다.
- 결과 요약 메시지를 제공해야 한다.

### 10.7 최근 이력

- 최근 변환 이력을 앱 재실행 후에도 유지해야 한다.
- 최근 이력은 최근 10건까지 보관해야 한다.
- 각 이력에는 원본 파일명, 출력 폴더, 요약 정보를 저장해야 한다.
- 최근 파일 경로와 최근 출력 폴더도 함께 저장 및 복원되어야 한다.

### 10.8 빌드 및 배포 폴더 구성

- 릴리스 빌드 결과는 사용자가 이해하기 쉬운 단순한 폴더 구조로 제공되어야 한다.
- 최종 배포 폴더에는 실행에 필요한 파일만 포함해야 한다.
- 개발용 로그, 임시 파일, 중간 빌드 산출물은 최종 배포 폴더에서 제외해야 한다.
- 가능하면 단일 루트 폴더 기준으로 실행 파일, 필수 DLL, 설정 파일이 정리되어야 한다.
- 개발 빌드와 배포 빌드는 루트 `bin` 아래에서 `Debug`, `Release` 폴더로 명확히 구분되어야 한다.

## 19. 권장 MVP 정의

현재 MVP는 Windows x64용 WPF 애플리케이션으로서 다음 범위를 포함한다.

- PDF 및 이미지 선택
- 출력 형식 선택 (`PPTX`, `DOCX`, `DOC`)
- 출력 폴더 선택
- 진행 상태, 결과 파일 목록, 경고 표시
- 최근 변환 이력 저장/복원
- OCR 사용 여부 및 엔진 선택 (이미지는 필수 적용)
- 샘플 PDF/이미지 기준 변환 검증
